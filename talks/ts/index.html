▓▓▓ Custom Hooks

• Extracting your networking or state management into its
own module can help make your code more maintainable by:
‣ Isolates complexity
‣ Clarify intention of usage by separating dependencies
‣ Helps separate different layers of responsibility
• Basically a constructor function to return an instance of
a service, not just a collection of methods and variables

██ To Avoid

▓▓▓ Casting with as

• Avoid casting with as
• Typescript is a structural type system so it is rarely
necessary and usually an indication that your types are
inconsistent. It's really a manual way of circumventing
the type compiler.
• Only to be used with functions like JSON.parse or fetch.

██ To Leverage

▓▓▓ Type Inference

• Try to avoid defining the types of things.
• Manually specifying types can feel a lot more necessary
when any is used a lot, but the less you can use it, the
more accurate your types are going to be, since you're
going to be letting the compiler find the real types of
things.

██ To Leverage

▓▓▓ Type Inference

e.g.:

type AToB<B = any> = { [key: string]: B }
const someMap: AToB = { field: "hi" }
someMap = { field: true, otherField: false }
someMap = { field: null }

/*
 * The type of `field` here is not inferred as `string`,
since you
 * defined the type of `someMap` without the type argument.
Now, `field` has a
 * type of `any`, and the `length` property is also `any`.
*/
someMap.field.length

// instead, use `unknown`

type AToB<B = unknown> = { [key: string]: B }
const someMap: AToB<string> = { field: "hi" }

// Now the type of `field` is appropriately `string`, and
`length` is `number`
someMap.field.length

callback parameter types
  const handleClick = (e:

    console.log(e);
  }

  return <Button onClick={handleClick}>Submit!<
}

// instead

const Component = () => {
  // `e` will now infer the type from `onClick`
  return <Button onClick={(e) =>{
    console.log(e);
  }}>Submit!<
}

██ To Avoid

▓▓▓ @ts-expect-error

• This basically disables the typescript compiler and
potentially lets an indefinite number of type errors
through in whatever region you disable tsc in
• A bunch of these in our codebase from trying to enable
more and more compiler restrictions, but we're actively
trying to remove them so if you encounter one please try
to remove the comment directive and fix the underlying
type error that is exposed
‣ Tip: if the type system says something needs to be
undefined or null or of a different type than initially
built for, change the definition of the function
signature or object type, instead of modifying the code
to remove the optional check.

// `someState` has a type of null and `setSomeState` only
can be passed `null`
const [someState, setSomeState] = useState(null)

// `count` cannot be instantiated with `null` if it has a
type of `number`
const [count, setCount] = useState<number>(null)

// instead...

// pass the type argument to `useState`
const [count, setCount] = useState<number | null>(null)
